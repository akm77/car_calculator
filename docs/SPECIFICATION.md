# **Техническое задание: Калькулятор ввоза легковых автомобилей в РФ (2025)**

## **1. Общие положения**

Калькулятор предназначен для расчёта **полной стоимости ввоза и оформления** легкового автомобиля категории **M1 (некоммерческое использование)** на территорию Российской Федерации из следующих стран:
- Япония  
- Республика Корея  
- ОАЭ  
- Китай  
- Грузия  

> **Не применяется** к коммерческим, грузовым, пассажирским, спецТС, электромобилям с иной категорией, пикапам и авто категории N. Для таких ТС — только ручной расчёт через поддержку.

---

## **2. Входные данные от пользователя**

Калькулятор должен запрашивать у пользователя:

| Поле | Тип | Комментарий |
|------|-----|-------------|
| Страна происхождения | Список | Япония, Корея, ОАЭ, Китай, Грузия |
| Год выпуска | Число | Определяет возраст авто и категорию пошлины |
| Стоимость автомобиля | Число + валюта | Для Японии — в JPY, для остальных — в USD или RUB (с конвертацией) |
| Объём двигателя | Число (см³) | Обязателен для расчёта пошлины и текущего утильсбора |
| Мощность двигателя | Число (л.с.) | Обязательно запрашивается — для будущих расчётов и внутреннего перевода в кВт |
| Тип транспортировки | Выбор (только для ОАЭ) | Открытый / Контейнер |
| Санкционный статус | Чекбокс (только для Японии) | Если не уверен — подсказка: «Обратитесь в поддержку» |

> **Примечание:**  
> Внутренне: `Мощность (кВт) = Мощность (л.с.) × 0.7355` — сохраняется для возможного перехода на новую систему утильсбора (по мощности).

#### **4.3.2. Утилизационный сбор (УС)**

**✅ Статус: Реализовано в v2.0 (2025-12-08)**

Новая система утилизационного сбора с двумерной таблицей (объём + мощность).

**Входные данные:**
- Объём двигателя (см³)
- **Мощность двигателя (л.с.)** ← НОВОЕ ОБЯЗАТЕЛЬНОЕ ПОЛЕ
- Возраст автомобиля

**Расчёт:**
1. Конвертация л.с. → кВт: `kw = hp × 0.7355`
2. Поиск диапазона объёма (5 bands)
3. Поиск диапазона мощности в выбранном band
4. Выбор коэффициента (lt3 или gt3)
5. Расчёт: `fee = 20,000 ₽ × coefficient`

**Конфигурация:** `config/rates.yml` (utilization_m1_personal)

**Формула**:  
\[
УС = Базовая\ ставка \times Коэффициент
\]

- **Базовая ставка (М1, физ.лицо)**: **20 000 руб.**

> **Важно:**
> - Объём и мощность запрашиваются **для расчета по новой системе** (Минпромторг: с 2025 г. планируется переход на расчёт по мощности).  
> - Для коммерческих авто — не рассчитывается.

Спасибо за уточнение. Ниже — **точная, компактная и программно применимая версия утилизационного сбора для физических лиц при ввозе авто для личного пользования**, **действующая с 1 декабря 2015 г.**, согласно вашей таблице.

## ✅ **Как использовать в калькуляторе**

1. Пользователь вводит:
   - Объём двигателя (см³)
   - Мощность (л.с.) → конвертируется в кВт: `кВт = л.с. × 0.7355`
   - Год выпуска → определяет возраст (≤3 или >3 лет на дату расчёта)

2. Система:
   - Находит строку по объёму
   - Находит поддиапазон по мощности (кВт)
   - Выбирает коэффициент в зависимости от возраста
   - Считает: `УС = 20 000 × Коэффициент`

### **4.5. Банковская комиссия (надбавка к валютному курсу)**

> Этот подпункт описывает **спецификацию** поведения. В актуальной реализации
> банковская комиссия уже применяется как надбавка к валютному курсу для всех
> релевантных валютных операций, включая конвертацию фиксированной комиссии
> компании 1000 USD в RUB.

#### 4.5.1. Определение

- **Банковская комиссия** — это *процентная надбавка к валютному курсу*,
  применяемая к операциям в иностранной валюте **до** конвертации в RUB.
- Базовая формула (логический контракт):

  - `effective_rate = base_rate × (1 + bank_commission_percent / 100)`

- Где:
  - `base_rate` — курс валюты к RUB из конфигурации/источника курсов
    (`rates.yml` +, опционально, CBR);
  - `bank_commission_percent` — глобальный процент банковской комиссии
    (из конфигурации комиссий);
  - `effective_rate` — итоговый курс, фактически использованный при
    расчёте сумм в RUB.

> Эффект банковской комиссии **не выводится** отдельной строкой в API и UI.
> Он отражается только через:
> - изменённое значение `breakdown.total_rub` (итоговая сумма),
> - содержимое `CalculationMeta.rates_used` /
>   `CalculationMeta.detailed_rates_used` (см. ниже).

#### 4.5.1.1. Влияние на `CostBreakdown` и `total_rub`

**Контракт:**

- Структура объекта `CostBreakdown` в JSON **остаётся неизменной**.  
  В частности, **не** добавляются поля вида `bank_commission_percent`,
  `bank_commission_rub` и т.п.
- Все компоненты `CostBreakdown`, которые зависят от валютного курса, внутри
  движка рассчитываются по `effective_rate`, а не по «сырому» `base_rate`:

  - `purchase_price_rub` — цена авто в RUB после конвертации стоимости покупки
    из валюты (`USD`, `JPY`, …) по `effective_rate`;
  - `freight_rub` — фрахт, конвертированный по `effective_rate`, если базовая
    ставка задана в валюте;
  - `country_expenses_rub` — расходы в стране покупки, если они заданы в
    валюте, после конвертации по `effective_rate`;
  - `company_commission_rub` — комиссия компании, рассчитанная как 1000 USD
    (или иное значение из `commissions.yml`) и конвертированная по
    `effective_rate`;
  - при необходимости — любые иные поля breakdown, зависящие от валютного курса.

- Итоговая сумма:

  - `total_rub` = сумма всех компонент `CostBreakdown` (включая те, что
    рассчитаны по `effective_rate`).  
  - Следовательно, `total_rub` **уже включает эффект банковской комиссии**,
    но сама комиссия не выделена как отдельная сумма в JSON.

> Клиенты API не должны пытаться вычислять «сумму банковской комиссии» из
> ответа. Эффект комиссии виден только как увеличение рублёвых сумм и как
> надбавка `+ PERCENT%` в строке курса (см. ниже). Структура JSON стабильна.

#### 4.5.2. Структура конфигурации `bank_commission`

Логическое местоположение — файл `config/commissions.yml`, отдельная секция `bank_commission`.

Рекомендуемая структура (пример):

```yaml
bank_commission:
  enabled: true            # опционально; если false или отсутствует, комиссия считается 0%
  percent: 2.0             # глобальная надбавка к курсу, в процентах

  meta:
    recommended_min: 0.0   # рекомендованный минимум
    recommended_max: 10.0  # рекомендованный максимум (мягкое ограничение)
    warn_above: 10.0       # порог, выше которого конфиг в будущем должен помечаться warning'ом
    default_percent: 0.0   # значение по умолчанию, если percent не задан

  # Возможное будущее расширение (в этом ТЗ фиксируется только как опция дизайна)
  # by_component:
  #   car_purchase_usd:
  #     percent: 2.0
  #   freight_usd:
  #     percent: 1.0
  #   local_costs_jpy:
  #     percent: 3.5
```

**Требования к поведению по умолчанию:**

- Если секция `bank_commission` **отсутствует целиком**, банковская комиссия считается равной `0%`.
- Если `bank_commission` есть, но:
  - `enabled: false` — комиссия считается `0%` независимо от `percent`;
  - `percent` отсутствует — используется `default_percent` из `meta` (если он не задан, трактуется как `0%`).

**Рекомендуемый диапазон значений:**

- `0–10%` — рекомендуемый «зелёный» диапазон.
- `>10%` — допускается, но в будущем реализация должна:
  - логировать предупреждение;
  - и/или добавлять соответствующее предупреждение в метаданные расчёта (см. ниже).

#### 4.5.3. Точки применения банковской комиссии

- Банковская комиссия применяется к **операциям в иностранной валюте** до конвертации в RUB:
  - покупка автомобиля в USD/JPY и др.;
  - расходы по стране (inspection, inland, local fees и т.п.), если они заданы в валюте;
  - фрахт, если он номинирован в валюте;
  - **комиссия компании** (фиксированная 1000 USD или переопределения по странам) — при конвертации суммы комиссии из USD в RUB используется `effective_rate` с учётом банковской комиссии.
- Для всех таких операций должен использоваться `effective_rate`, а не «сырой» `base_rate`.

#### 4.5.4. Связь с метаданными `CalculationMeta` и `meta.rates_used`

Модель `CalculationMeta` (см. `app/calculation/models.py`) содержит поля
`rates_used` и `detailed_rates_used`, в которых трекаются используемые
курсы валют.

Требование к структуре на уровне модели (внутренний контракт движка):

- Для каждой используемой валюты (`USD`, `JPY` и т.п.) в `rates_used` должны
  быть доступны как минимум следующие **логические** поля:

  - `base_rate` — исходный курс из конфигурации/источника;
  - `bank_commission_percent` — применённый процент банковской комиссии
    (0, если отключено);
  - `effective_rate` — итоговый курс после применения банковской комиссии;
  - `display` — подготовленная строка для UI (`"USD/RUB = BASE [+ PERCENT%]"`).

Пример целевой структуры (модельное представление, **не является прямым
JSON-контрактом публичного API**):

```json
{
  "rates_used": {
    "USD_RUB": {
      "base_rate": 78.95,
      "bank_commission_percent": 1.0,
      "effective_rate": 79.7395,
      "display": "USD/RUB = 78.95 + 1%"
    },
    "JPY_RUB": {
      "base_rate": 0.52,
      "bank_commission_percent": 0.0,
      "effective_rate": 0.52,
      "display": "JPY/RUB = 0.52"
    }
  }
}
```

> В этом спринте **структура JSON‑ответа `/api/calculate` не меняется**:
> публичный контракт по-прежнему определён полями `breakdown`, `meta` и
> `request`, как описано выше. Структура `rates_used` рассматривается как
> внутреннее представление движка и может быть сериализована в упрощённой форме.

##### 4.5.4.1. Минимальный контракт для клиентов через `meta.rates_used`

Чтобы WebApp и Telegram‑бот могли корректно отобразить курс и наличие
банковской комиссии, достаточно, чтобы сервер предоставил для основной
валютной пары расчёта **строковое представление курса** вида:

- `"USD/RUB = BASE_RATE + PERCENT%"` — если `bank_commission_percent > 0`;
- `"USD/RUB = BASE_RATE"` — если банковская комиссия отсутствует или равна 0.

Формально:

- В JSON‑ответе `/api/calculate` поле `meta.rates_used` (или
  совместимое поле с другим именем) **должно** содержать хотя бы одну запись
  для базовой валюты расчёта (например, `USD_RUB` или `JPY_RUB`).
- Для клиентских приложений **официально гарантируется** минимум:

  - наличие строкового поля (например, `display`), пригодного для прямого
    показа пользователю;
  - формат строки:

    ```text
    <BASE_CURRENCY>/RUB = <BASE_RATE>[ + <PERCENT>%]
    ```

    где часть `+ <PERCENT>%` присутствует только при ненулевой
    банковской комиссии.

- Клиентам **не требуется** знать внутренние поля `base_rate`,
  `bank_commission_percent`, `effective_rate` и формулу их вычисления —
  они могут опираться исключительно на готовую строку.

##### 4.5.4.2. Обратная совместимость

- Старые клиенты, которым нужны только числовые поля `breakdown.*`, могут
  полностью игнорировать `meta.rates_used` — семантика `total_rub` при этом
  остаётся корректной (она просто включает эффект банковской комиссии).  
- При `bank_commission_percent = 0` строка курса не содержит суффикса `+ X%`,
  оставаясь в виде `USD/RUB = BASE_RATE`.

#### 4.5.5. Различие между комиссией компании и банковской комиссией

Важно явно различать два вида комиссий:

- **Комиссия компании** (уже реализована):
  - фиксированная величина в валюте (сейчас 1000 USD, с исключениями для отдельных стран);
  - задаётся в `config/commissions.yml` как `default_commission_usd` и `by_country.*.commission_usd`;
  - представляет собой сервисный сбор посредника/компании.

- **Банковская комиссия** (находится на стадии спецификации в этом документе):
  - процентная надбавка к валютному курсу (`bank_commission.percent`),
  - применяется ко всем релевантным конвертациям «валюта → RUB» через `effective_rate`,
  - влияет на `total_rub` и `meta.rates_used`, но не отображается отдельной строкой в breakdown.

## ✅ Как использовать в калькуляторе (обновление в части commission)

1. Пользователь вводит:
   - Объём двигателя (см³)
   - Мощность (л.с.) → конвертируется в кВт: `кВт = л.с. × 0.7355`
   - Год выпуска → определяет возраст (≤3 или >3 лет на дату расчёта)

2. Система:
   - Находит строку по объёму
   - Находит поддиапазон по мощности (кВт)
   - Выбирает коэффициент в зависимости от возраста
   - Считает: `УС = 20 000 × Коэффициент`

3. **Комиссия компании:**
   - Всегда **1000 USD** (для всех стран, кроме ОАЭ).
   - Для ОАЭ — комиссия **0 USD** (включена в расходы страны).

4. **Банковская комиссия:**
   - Применяется как надбавка к валютному курсу для всех операций в иностранной валюте.
